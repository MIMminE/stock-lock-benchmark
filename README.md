# 단일 Row 핫스팟 재고 차감 벤치마크 보고서 (동시성 스윕 5/10/30/50/100)

## 1) 실험 목적

본 실험의 목적은 **단일 재고 row(핫스팟)** 에 대해 다수의 동시 요청이 **재고 차감(update)** 을 수행할 때,

- **낙관적 락(Optimistic Lock, 충돌 시 재시도)** 과
- **비관적 락(Pessimistic Lock, 락 대기 기반 직렬화)**

두 전략이 **락 경합을 처리하는 방식 차이**로 인해 어떤 성능 특성(처리량/지연/tail/재시도 비용)을 보이는지 정량적으로 비교하는 것이다.

---

## 2) 실험 설계 및 파라미터

### 2.1 공통 설정(5개 실험 동일)

- 초기 재고: `initialStock = 20000`
- 목표 성공 건수: `targetSuccessCount = 20000` (재고를 0까지 소진)
- backoff: `backoffMillis = 5ms`
- 재시도 한도: `maxRetriesPerSuccess = 500`
- 각 동시성(concurrency)마다 **Optimistic → Pessimistic** 두 Phase를 측정

### 2.2 동시성 스윕(독립 변수)

- concurrency = **5 / 10 / 30 / 50 / 100**

### 2.3 실패(`return_false`)의 의미(중요)

모든 실험에서 `totalRequests > targetSuccessCount` 이고 `failuresByReason.return_false`가 존재합니다.
이는 보통,

- 목표 성공(20000)에 도달하는 순간에도 일부 요청이 **이미 in-flight** 상태이며
- 그 요청들이 재고 0 상태를 만나 “정상적으로 차감 실패(false)”로 종료되는
  **종료 조건이 만든 자연스러운 거절**로 해석하는 것이 타당합니다.  
  (즉, 시스템 오류라기보다는 테스트 러너의 종료/경쟁 상황에서 발생하는 정상 케이스)

성공률은 $$\text{successRate}=\frac{\text{successCount}}{\text{totalRequests}}$$ 로 계산됩니다.

## 4) 비교 분석

### 4.1 처리량(Throughput): 동시성 증가에 따른 “격차 확대”

비관적 락이 낙관적 락 대비 처리량이 얼마나 높은지(배수):

| Concurrency | $$\frac{TP_{pess}}{TP_{opt}}$$ |
|------------:|-------------------------------:|
|           5 |                          1.28× |
|          10 |                          1.61× |
|          30 |                          2.83× |
|          50 |                          2.92× |
|         100 |                          3.55× |

**해석**

- concurrency가 낮을 때(5~10)는 낙관락도 재시도가 “관리 가능한 수준”이라 격차가 상대적으로 작습니다.
- concurrency가 30 이상으로 올라가면 단일 row 충돌 확률이 급증하며, 낙관락은 **재시도 + backoff 누적 비용**이 본격적으로 성능을 갉아먹습니다.
- 비관락은 락 대기(큐잉)로 경합을 흡수하되, 재시도 폭발이 없어 **처리량이 상대적으로 안정적으로 유지**됩니다.

### 4.2 지연시간(Latency): 평균도 tail도 낙관락이 급격히 악화

평균 성공 지연시간의 비율(낙관/비관):

| Concurrency | $$\frac{Lat_{opt}}{Lat_{pess}}$$ |
|------------:|---------------------------------:|
|           5 |                            1.26× |
|          10 |                            1.60× |
|          30 |                            2.85× |
|          50 |                            2.93× |
|         100 |                            3.55× |

Tail(p99)도 대부분 구간에서 낙관락이 크게 불리합니다. 예를 들어 concurrency=100에서

- optimistic p99 = **4677ms**
- pessimistic p99 = **1206ms**

즉 $$\frac{4677}{1206}\approx 3.88$$ 로 tail이 약 3.9배 수준까지 벌어졌습니다.

### 4.3 재시도 비용: 낙관락의 병목이 숫자로 드러남

낙관락은 동시성 증가에 따라 평균 시도 횟수가 다음과 같이 상승합니다.

- concurrency=5: **2.23 attempts/success**
- concurrency=30: **9.09 attempts/success**
- concurrency=100: **11.94 attempts/success**

이는 단일 핫스팟에서 업데이트 충돌이 반복되며, 성공 1건을 만들기 위해 “거의 12번” 정도의 시도가 필요해졌다는 의미입니다.  
결과적으로 DB 입장에서는 **불필요한 업데이트 시도/롤백/재조회**가 폭증하고, 애플리케이션 입장에서는 **backoff 대기**가 누적되어 처리량과 tail이 동시에 악화됩니다.

### 4.4 (특이점) concurrency=10에서 낙관락 p95가 낮지만 p99가 매우 큼

concurrency=10의 성공 지연:

- optimistic p95=17ms, p99=924ms

이는 흔히 “대부분은 빠른데, 일부 요청이 충돌 연속으로 길게 늘어지는” **긴 꼬리(long tail)** 패턴입니다.  
낙관락은 충돌이 ‘희귀’할 때 평균/중앙값이 좋아 보일 수 있으나, 핫스팟에서는 tail이 급격히 나빠질 수 있음을 보여줍니다.

---

## 5) 결론

- 단일 row 핫스팟 재고 차감에서 동시성이 커질수록, 낙관적 락은 충돌로 인한 **재시도/백오프 비용이 비선형적으로 증가**하며 처리량과 tail latency가 함께 악화되었다.
- 동일 조건에서 비관적 락은 락 대기를 통해 경합을 직렬화하여 **재시도 비용 없이 더 안정적인 처리량/지연**을 유지했고, concurrency=100에서는 처리량이 약 **3.55배** 높게 측정되었다.

## 6) 한계 및 공정성(Validity & Fairness)

### 핫스팟 전제의 한계

본 실험은 다품목으로 트래픽이 분산되는 상황이 아니라, **단일 상품(단일 row)** 에 요청이 집중되는 **핫스팟**을 전제로 한다.  
따라서 결과를 “일반적인 전체 트래픽에서의 우열”로 **일반화해서 해석하면 안 된다.**

### 환경 의존성 명시 필요

관측된 처리량/지연시간은 애플리케이션·DB 리소스에 크게 좌우된다. 특히 아래 설정에 따라 결과가 달라질 수 있으므로, 보고서에는 **실제 설정값을 함께 명시**하는 것이 공정하다.

- HikariCP 풀 크기
- 서버 스레드(톰캣 등)
- DB CPU/IO
- 트랜잭션 격리 수준 등

### `return_false`의 의미(정상 결과로 분류)

`return_false`는 시스템 오류가 아니라, **재고가 이미 소진된 뒤 도착한 in-flight 요청**이 “구매 실패”로 정상 종료된 결과로 간주한다.

따라서 비교의 핵심은 **실패율 자체**가 아니라,

- **성공 처리 성능**(throughput/latency)
- **재시도 비용**(retry, 충돌, 롤백 비용)
  에 둔다.

## 7) 다음 실험 제안

### 1. 재시도(backoff) 정책 비교

고정 5ms vs **지수 backoff + jitter**를 비교해, 충돌 구간에서의 **p95/p99 지연** 및 **시도 횟수(AvgAttempts)** 변화를 측정한다.

### 2. 커넥션 풀 파라미터 스윕

concurrency를 고정(예: 50/100)하고 **Hikari pool size**를 변화시키며 병목이 “락”인지 “커넥션/스레드”인지 분리해 확인한다.

### 3. 키 분산(다품목) 시나리오 실험

상품을 N개로 분산해 충돌률이 낮아질 때, **낙관락의 장점이 살아나는 구간**(성공률/지연/시도 횟수 개선)을 확인한다.

### 4. 핫스팟 해소 설계 비교

아래 설계를 비교해 **락 경쟁 자체를 줄이는 접근**의 효과를 평가한다.

- **카운터/재고 샤딩**: 재고를 N개 row로 분산
- **큐 기반 직렬 처리**: 단일 키에 대한 업데이트를 순서화
