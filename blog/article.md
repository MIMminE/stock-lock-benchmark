# Stock Lock Benchmark

재고 차감 로직을 구현할 때 자주 나오는 질문이 있습니다. "낙관락이 좋을까, 비관락이 좋을까?" 정답은 상황에 따라 다릅니다. 하지만 특정 상품 하나에 주문 요청이 몰리는 핫스팟 상황이라면 이야기가 달라집니다.

Stock Lock Benchmark는 단일 재고 row에 동시 주문 요청이 집중되는 상황을 만들고, Optimistic Lock과 Pessimistic Lock의 처리량, 지연시간, 재시도 비용을 비교한 실험형 포트폴리오입니다. 이 프로젝트는 멋진 화면을 만드는 것보다, 동시성 제어 전략을 선택할 때 어떤 데이터를 봐야 하는지를 보여주는 데 집중했습니다.

## 이 프로젝트에서 확인할 수 있는 역량

- 단일 row 핫스팟 상황을 가정한 재고 차감 동시성 실험 설계
- Optimistic Lock과 Pessimistic Lock의 처리량, latency, retry 비용 비교
- 평균 latency뿐 아니라 p95/p99와 성공당 시도 횟수를 함께 보는 결과 해석
- Spring Boot, PostgreSQL, Micrometer 기반 실험 환경 구성
- 실험 결과를 콘솔과 리포트 화면으로 정리해 의사결정 근거로 설명하는 방식

## 문제 상황

커머스 서비스에서 재고 차감은 단순해 보입니다.

```text
상품 조회
-> 재고 확인
-> 재고 차감
-> 주문 생성
```

하지만 인기 상품, 한정 수량 이벤트, 라이브 커머스처럼 특정 상품 하나에 요청이 몰리면 재고 row 하나가 병목이 됩니다. 이때 낙관락은 충돌이 발생한 요청을 재시도해야 하고, 비관락은 요청을 DB row lock 대기로 직렬화합니다.

둘 중 어떤 방식이 더 낫다고 단정하기보다, 같은 조건에서 실제로 어떤 비용이 발생하는지 비교해보고 싶었습니다.

## 해결하고 싶었던 문제

이 프로젝트는 “낙관락과 비관락을 모두 써봤다”를 보여주기 위한 예제가 아닙니다. 백엔드 설계 리뷰나 면접에서 락 전략을 선택할 때, 어떤 지표와 전제를 가지고 판단해야 하는지 보여주는 것이 목적입니다.

| 질문 | 실험에서 확인한 것 |
| --- | --- |
| 낙관락은 충돌이 많아질 때 어떤 비용을 만드는가 | retry 횟수, 성공당 평균 시도 횟수, p95/p99 latency |
| 비관락은 무조건 느린가 | 단일 row 핫스팟에서는 retry 폭발 없이 예측 가능한 대기를 만드는지 확인 |
| 평균 latency만 봐도 충분한가 | tail latency가 사용자 경험과 장애 판단에 더 중요할 수 있음을 확인 |
| 락 전략을 일반화해도 되는가 | 단일 row 핫스팟이라는 실험 조건과 한계를 명시 |
| 결과를 어떻게 의사결정 자료로 바꿀 것인가 | 콘솔, 요약 그래프, 보고서 화면으로 실험 조건과 해석을 함께 제공 |

## 실험 설계

실험은 단일 상품의 재고 row 하나에 여러 동시 요청을 보내는 방식으로 구성했습니다.

| 조건 | 값 |
| --- | --- |
| 초기 재고 | `20000` |
| 목표 성공 건수 | `20000` |
| Backoff | `5ms` |
| 성공당 최대 재시도 | `500` |
| Phase 순서 | `optimistic -> reset -> pessimistic` |
| 동시성 | `5 / 10 / 30 / 50 / 100` |

관찰한 지표는 단순 성공 여부가 아닙니다. 처리량, 평균 지연시간, p95, p99, 성공당 평균 시도 횟수, 전체 retry 수를 함께 봤습니다.

## 실험 콘솔

![Benchmark console](./images/01-benchmark-console.png)

콘솔에서는 실험 조건, phase별 실행 상태, 처리량, latency, retry 비용을 한 화면에서 확인할 수 있습니다. API 결과만 보는 것보다 실험 과정을 시각적으로 확인할 수 있게 한 이유는, 이 프로젝트가 "벤치마크 보고서"이면서 동시에 포트폴리오 게시물이기 때문입니다.

## 주요 결과

![Benchmark summary](./images/02-benchmark-summary.svg)

| Concurrency | Phase | Duration(s) | Throughput(success/s) | Avg Success Lat(ms) | p95(ms) | p99(ms) | Avg Attempts/Success | Total Retries |
| ---: | --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| 5 | optimistic | 53.540 | 373.552 | 12.741 | 69 | 244 | 2.232 | 24,668 |
| 5 | pessimistic | 41.752 | 479.019 | 10.104 | 32 | 68 | 1.000 | 0 |
| 30 | optimistic | 127.798 | 156.497 | 190.758 | 635 | 1,013 | 9.092 | 162,138 |
| 30 | pessimistic | 45.129 | 443.174 | 66.970 | 205 | 322 | 1.000 | 0 |
| 100 | optimistic | 178.095 | 112.300 | 880.411 | 2,886 | 4,677 | 11.942 | 221,179 |
| 100 | pessimistic | 50.144 | 398.851 | 248.201 | 756 | 1,206 | 1.000 | 0 |

결과를 보면 동시성이 낮을 때도 낙관락은 retry 비용이 발생합니다. 동시성이 30 이상으로 올라가면 충돌 확률이 급격히 커지고, 낙관락은 업데이트 실패, 재조회, backoff가 누적되면서 처리량과 tail latency가 함께 나빠집니다.

반면 비관락은 요청을 DB row lock 대기로 직렬화하기 때문에 retry 폭발이 없습니다. 단일 row 핫스팟이라는 제한된 조건에서는 더 안정적인 처리량과 지연시간을 보였습니다.

## 리포트 화면

![Benchmark report](./images/03-benchmark-report.png)

리포트 화면은 결과 표만 복사해 둔 문서가 아니라, 실험 조건과 해석을 함께 확인하는 화면입니다. 포트폴리오를 보는 사람이 "왜 이 수치를 봐야 하는지"를 바로 이해할 수 있도록 실험 조건, 결과, 한계를 같은 흐름으로 배치했습니다.

## 결과를 어떻게 해석할까

이 실험이 "비관락이 항상 더 좋다"는 결론을 말하는 것은 아닙니다. 상품이 여러 row로 분산되고 충돌 확률이 낮은 일반 트래픽에서는 낙관락이 더 나은 구간이 있을 수 있습니다.

하지만 특정 상품에 요청이 집중되는 상황이라면 retry 비용이 생각보다 빠르게 커집니다. 이 프로젝트에서는 그 비용을 감이 아니라 수치로 확인했습니다.

특히 면접이나 설계 리뷰에서 중요한 포인트는 다음입니다.

- 락 전략은 기술 취향이 아니라 트래픽 패턴에 따라 달라진다.
- 평균 latency만 보면 안 되고 p95, p99를 같이 봐야 한다.
- 낙관락의 retry는 애플리케이션 비용과 DB 재시도 비용을 함께 만든다.
- 비관락은 대기를 만들지만, 핫스팟에서는 예측 가능한 대기일 수 있다.

## 구현 포인트

이 프로젝트는 Spring Boot와 PostgreSQL을 사용해 두 전략을 같은 조건에서 비교합니다.

```text
Optimistic Lock
-> version 컬럼 충돌 감지
-> 실패 시 backoff 후 재시도

Pessimistic Lock
-> SELECT ... FOR UPDATE
-> row lock 대기 후 순차 차감
```

실험 결과는 Micrometer 지표와 콘솔 화면으로 확인할 수 있게 구성했습니다. Prometheus, Grafana를 붙여 관측 가능한 실험 환경으로 확장할 수 있도록 했습니다.

## 설계 판단

| 설계 선택 | 이유 |
| --- | --- |
| 단일 stock row 핫스팟 | 인기 상품, 한정 수량 이벤트처럼 충돌이 가장 크게 드러나는 조건을 만들기 위해 |
| 목표 성공 건수 고정 | 각 전략이 같은 성공량을 처리하는 동안 발생한 시간과 retry 비용을 비교하기 위해 |
| p95/p99 포함 | 평균값만으로는 사용자가 체감하는 지연과 tail risk를 설명하기 어렵기 때문에 |
| 성공당 시도 횟수 기록 | 낙관락 retry 비용을 처리량과 함께 해석하기 위해 |
| 콘솔/보고서 UI 제공 | 실험 결과를 단순 로그가 아니라 의사결정 자료처럼 읽을 수 있게 하기 위해 |

이 프로젝트는 작은 벤치마크지만, 백엔드 개발자가 동시성 문제를 감이 아니라 실험 조건과 지표로 설명할 수 있다는 점을 보여주기 위한 포트폴리오입니다.

## 회고

이 프로젝트는 큰 시스템은 아니지만, 백엔드 면접에서 자주 나오는 동시성 질문을 가장 직접적으로 보여줄 수 있는 포트폴리오입니다.

중요한 것은 "낙관락을 써봤다", "비관락을 써봤다"가 아니라 어떤 조건에서 어떤 비용이 발생하는지 설명할 수 있는가입니다. Stock Lock Benchmark는 그 설명을 숫자, 그래프, 콘솔 화면으로 뒷받침하기 위해 만든 프로젝트입니다.

## 면접에서 설명할 포인트

- 낙관락과 비관락의 선택 기준은 기술 취향이 아니라 충돌 확률과 트래픽 패턴이라는 점
- 단일 row 핫스팟에서는 retry 비용이 처리량과 tail latency를 동시에 악화시킬 수 있다는 점
- 비관락은 대기를 만들지만, 특정 조건에서는 더 예측 가능한 처리 흐름을 제공할 수 있다는 점
- 평균 latency뿐 아니라 p95/p99, retry 수, 성공당 시도 횟수를 함께 봐야 한다는 점
- 실험 결과를 운영 의사결정 자료로 바꾸려면 조건, 한계, 해석을 함께 남겨야 한다는 점
